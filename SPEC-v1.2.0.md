# Harbor v1.2.0 — Complete Implementation Spec
## Poll-Based Companion Architecture + HA Update Fix

---

## Overview

This is a breaking architectural change to how Harbor communicates with the Harbor Companion add-on.

**The problem with v1.1.x:** Harbor called INTO the companion (push model), which required the companion to be publicly reachable via a specific URL, exposed port, and Cloudflare path routing. This broke constantly and made remote HA instances nearly impossible to set up.

**The fix in v1.2.0:** Flip the direction. The companion POLLS Harbor for commands (pull model). Harbor is already publicly reachable. The companion only needs outbound HTTPS — which always works from inside any HA instance, anywhere in the world.

**Inspiration:** Portainer's Edge Agent uses the exact same pattern and scales to 15,000+ remote agents.

## Part 1: Database Changes

Add these two new tables to the SQLite schema (in `src/database.js` or wherever `CREATE TABLE` statements live).

### Table: `companion_tokens`
Stores one-time setup tokens generated by Harbor for companion onboarding.

```sql
CREATE TABLE IF NOT EXISTS companion_tokens (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  instance_id INTEGER NOT NULL,
  token TEXT NOT NULL UNIQUE,
  expires_at DATETIME NOT NULL,
  used INTEGER DEFAULT 0,
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (instance_id) REFERENCES instances(id) ON DELETE CASCADE
);
```

### Table: `companion_commands`
The command queue. Harbor writes commands here; companion picks them up and writes results back.

```sql
CREATE TABLE IF NOT EXISTS companion_commands (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  instance_id INTEGER NOT NULL,
  command TEXT NOT NULL,
  payload TEXT DEFAULT NULL,
  status TEXT DEFAULT 'pending',
  result TEXT DEFAULT NULL,
  error TEXT DEFAULT NULL,
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  completed_at DATETIME DEFAULT NULL,
  FOREIGN KEY (instance_id) REFERENCES instances(id) ON DELETE CASCADE
);
```

### Modify: `instances` table
Add/ensure these columns exist (use `ALTER TABLE ... ADD COLUMN IF NOT EXISTS` pattern or handle in migration):

```sql
ALTER TABLE instances ADD COLUMN companion_enabled INTEGER DEFAULT 0;
ALTER TABLE instances ADD COLUMN companion_last_seen DATETIME DEFAULT NULL;
ALTER TABLE instances ADD COLUMN companion_version TEXT DEFAULT NULL;
```

Remove `companion_url` and `companion_secret` from the instances table if they exist — these are no longer needed. Handle migration gracefully (check if column exists before dropping).

## Part 2: Backend API Changes

All routes go in `src/routes/` or wherever API routes are currently defined.

### 2.1 Remove Old Companion Routes

### 2.2 New Companion API Endpoints

#### `POST /api/companion/token/:instanceId`
**Called by:** Harbor frontend (when admin clicks "Generate Setup Token")  
**Purpose:** Generate a one-time setup token for the companion to use during onboarding

Logic:
1. Verify the instance exists
2. Delete any existing unused tokens for this instance
3. Generate a random 32-byte hex token
4. Create a JSON payload: `{ harbor_url, instance_id, secret }` where:
   - `harbor_url` = the public URL of Harbor itself (read from an env var `HARBOR_PUBLIC_URL` or from config, e.g. `https://harbor.blackcoffeenetworks.com`)
   - `instance_id` = the integer instance ID
   - `secret` = a newly generated random 32-byte hex string (this becomes the shared secret for all future companion communication for this instance — store it in the instances table as `companion_secret`)
5. Base64-encode the JSON payload → this is the setup token
6. Store token in `companion_tokens` with `expires_at = NOW + 24 hours`
7. Return: `{ token, expires_at }`

**Note:** Also add `HARBOR_PUBLIC_URL` as a required environment variable. Add it to the Docker Compose / container config template.

---

#### `POST /api/companion/register`
**Called by:** Companion on first startup with the setup token  
**Purpose:** Complete onboarding — companion exchanges setup token for its config

Request body:
```json
{
  "setup_token": "base64encodedstring",
  "companion_version": "1.2.0"
}
```

Logic:
1. Base64-decode `setup_token` → parse JSON → extract `{ harbor_url, instance_id, secret }`
2. Look up token in `companion_tokens` where `token = setup_token AND used = 0 AND expires_at > NOW`
3. If not found → return 401
4. Mark token as `used = 1`
5. Store `secret` in `instances` table for this `instance_id` as `companion_secret`
6. Set `companion_enabled = 1` on the instance
7. Update `companion_last_seen = NOW` and `companion_version`
8. Return: `{ success: true, instance_id, poll_interval_seconds: 10 }`

---

#### `GET /api/companion/poll/:instanceId`
**Called by:** Companion every 10 seconds  
**Auth:** `X-Harbor-Secret` header must match `companion_secret` stored for this instance  
**Purpose:** Companion asks "do you have any commands for me?"

Logic:
1. Verify auth header matches `companion_secret` for the instance
2. Update `companion_last_seen = NOW` on the instance
3. Query `companion_commands` for the oldest `status = 'pending'` command for this instance
4. If found:
   - Set that command's `status = 'processing'`
   - Return: `{ command_id: id, command: command, payload: payload }`
5. If none:
   - Return: `{ command_id: null }`

---

#### `POST /api/companion/result/:commandId`
**Called by:** Companion after executing a command  
**Auth:** `X-Harbor-Secret` header  
**Purpose:** Companion posts the result of a command back to Harbor

Request body:
```json
{
  "instance_id": 1,
  "status": "done",
  "result": { ... },
  "error": null
}
```

Logic:
1. Verify auth header matches `companion_secret` for the given `instance_id`
2. Update `companion_commands` row:
   - `status = body.status` ('done' or 'error')
   - `result = JSON.stringify(body.result)`
   - `error = body.error`
   - `completed_at = NOW`
3. Return: `{ success: true }`

---

#### `POST /api/instances/:id/companion/command`
**Called by:** Harbor frontend (when user opens a tab that needs companion data)  
**Purpose:** Frontend enqueues a command and gets back a command ID to poll for the result

Request body:
```json
{
  "command": "GET_UPDATES"
}
```

Valid commands:
- `GET_UPDATES`
- `GET_ADDONS`
- `GET_BACKUPS`
- `GET_SYSTEM`
- `REBOOT_HOST`
- `RESTART_HA`
- `SHUTDOWN_HOST`
- `UPDATE_CORE`
- `UPDATE_SUPERVISOR`
- `UPDATE_OS`
- `UPDATE_ADDON` (payload: `{ addon_slug, version }`)
- `BACKUP_NOW` (payload: `{ name }`)

Logic:
1. Check `companion_enabled = 1` for this instance; if not, return 400 with message "Companion not connected"
2. Check `companion_last_seen` is within the last 60 seconds; if not, return 503 with message "Companion offline"
3. **For data commands (GET_*):** Check if there's already a `done` result for this command from the last 30 seconds — if so, return it immediately from cache (avoid re-queuing the same command repeatedly)
4. Insert new row into `companion_commands` with `status = 'pending'`
5. Return: `{ command_id: id }`

---

#### `GET /api/instances/:id/companion/result/:commandId`
**Called by:** Harbor frontend (polling for result after enqueuing a command)  
**Purpose:** Get the result of a previously enqueued command

Logic:
1. Fetch the `companion_commands` row by `id` (verify it belongs to this instance)
2. Return: `{ status, result, error, created_at, completed_at }`

Frontend should poll this every 2 seconds until `status` is `done` or `error`, with a 30-second timeout.

---

#### `GET /api/instances/:id/companion/status`
**Called by:** Harbor frontend  
**Purpose:** Check if companion is connected and get its info

Logic:
1. Return:
```json
{
  "enabled": true/false,
  "online": true/false,
  "last_seen": "ISO datetime or null",
  "version": "1.2.0 or null"
}
```
`online` = true if `companion_last_seen` is within last 60 seconds.

---

### 3.2 Updates Tab — Fix the Request/Response Flow

The Updates tab currently calls the old companion URL directly. Replace with the new queue-based approach:

```
On tab open/refresh:
1. Call POST /api/instances/:id/companion/command { command: "GET_UPDATES" }
2. Show a loading spinner
3. Poll GET /api/instances/:id/companion/result/:commandId every 2 seconds
4. On status=done → render the updates list
5. On status=error → show error message
6. If no result within 30 seconds → show timeout error

The result from the companion will be JSON like:
{
  "core": { "version": "2026.4.4", "version_latest": "2026.4.5", "update_available": true },
  "supervisor": { "version": "2026.04.1", "version_latest": "2026.04.1", "update_available": false },
  "os": { "version": "17.2", "version_latest": "17.2", "update_available": false }
}

If update_available is true for any item, show an "Update" button next to it.
Clicking Update → POST /api/instances/:id/companion/command { command: "UPDATE_CORE" } (or UPDATE_SUPERVISOR, UPDATE_OS)
Then show loading until result comes back.
```

## Part 4: Companion Changes (Python FastAPI)

All changes are in `addon/harbor-companion/app/main.py`.

### 4.1 New Config Schema

The companion now only needs ONE config field: `harbor_token`.

**On startup:**
1. Read `/data/options.json`
2. If `harbor_token` field exists and is non-empty:
   - Base64-decode it → parse JSON → extract `{ harbor_url, instance_id, secret }`
   - Store these in memory as the running config
   - Log: `INFO: Token decoded — will connect to {harbor_url} as instance {instance_id}`
3. If token field is empty → log error and exit with a helpful message

**Remove:** Reading `harbor_url`, `instance_id`, `harbor_secret`, `companion_url` from options.json. Those fields are gone.

### 4.2 Startup Registration

On startup (use the `lifespan` context manager, not the deprecated `on_event`):
1. Call `POST {harbor_url}/api/companion/register` with body:
   ```json
   { "setup_token": "<the raw base64 token>", "companion_version": "1.2.0" }
   ```
2. If response is 200: log success, extract `poll_interval_seconds` from response
3. If response is 401: log "Token already used or expired — generate a new token in Harbor" and continue running (the companion may already be registered from a previous start)
4. If response is any other error: log the error and continue

**Note:** On subsequent restarts after initial registration, the token is already `used = 1` in Harbor. That's fine — the companion still has the secret in its decoded token and can continue polling. The registration endpoint returning 401 on a used token should NOT stop the companion from running.

Actually, to handle restarts cleanly: save the decoded `{ harbor_url, instance_id, secret }` to `/data/companion_config.json` after the first successful registration. On subsequent startups, if this file exists, use it directly without calling register again. Only call register if this file doesn't exist (i.e., first-time setup).

### 4.3 Poll Loop

Start a background thread (or asyncio task) that runs the poll loop:

```python
async def poll_loop():
    while True:
        try:
            response = await http_client.get(
                f"{harbor_url}/api/companion/poll/{instance_id}",
                headers={"X-Harbor-Secret": secret},
                timeout=15
            )
            if response.status_code == 200:
                data = response.json()
                if data.get("command_id"):
                    await execute_command(data["command_id"], data["command"], data.get("payload"))
            elif response.status_code == 401:
                logger.error("Auth rejected by Harbor — check companion secret")
        except Exception as e:
            logger.warning(f"Poll failed: {e}")
        
        await asyncio.sleep(poll_interval_seconds)  # default 10
```

### 4.4 Command Execution

```python
async def execute_command(command_id, command, payload):
    result = None
    error = None
    status = "done"
    
    try:
        if command == "GET_UPDATES":
            result = await get_updates()
        elif command == "GET_ADDONS":
            result = await get_addons()
        elif command == "GET_BACKUPS":
            result = await get_backups()
        elif command == "GET_SYSTEM":
            result = await get_system_info()
        elif command == "REBOOT_HOST":
            result = await reboot_host()
        elif command == "RESTART_HA":
            result = await restart_ha()
        elif command == "SHUTDOWN_HOST":
            result = await shutdown_host()
        elif command == "UPDATE_CORE":
            result = await update_core()
        elif command == "UPDATE_SUPERVISOR":
            result = await update_supervisor()
        elif command == "UPDATE_OS":
            result = await update_os()
        elif command == "BACKUP_NOW":
            name = payload.get("name", "harbor-backup") if payload else "harbor-backup"
            result = await create_backup(name)
        else:
            error = f"Unknown command: {command}"
            status = "error"
    except Exception as e:
        error = str(e)
        status = "error"
    
    # Post result back to Harbor
    await http_client.post(
        f"{harbor_url}/api/companion/result/{command_id}",
        headers={"X-Harbor-Secret": secret},
        json={
            "instance_id": instance_id,
            "status": status,
            "result": result,
            "error": error
        }
    )
```

### 4.5 Supervisor API Helper Functions

Keep all existing Supervisor API calls (`get_updates`, `get_addons`, etc.) but fix the URL construction. They should call the Supervisor API directly using the `SUPERVISOR_TOKEN`:

```python
SUPERVISOR_URL = "http://supervisor"
SUPERVISOR_TOKEN = os.environ.get("SUPERVISOR_TOKEN", "")

async def supervisor_get(path):
    response = await http_client.get(
        f"{SUPERVISOR_URL}{path}",
        headers={"Authorization": f"Bearer {SUPERVISOR_TOKEN}"},
        timeout=30
    )
    response.raise_for_status()
    return response.json()

async def supervisor_post(path, data=None):
    response = await http_client.post(
        f"{SUPERVISOR_URL}{path}",
        headers={"Authorization": f"Bearer {SUPERVISOR_TOKEN}"},
        json=data,
        timeout=60
    )
    response.raise_for_status()
    return response.json()

async def get_updates():
    data = await supervisor_get("/supervisor/info")
    ha_data = await supervisor_get("/core/info")
    os_data = await supervisor_get("/os/info")
    return {
        "core": {
            "version": ha_data["data"]["version"],
            "version_latest": ha_data["data"]["version_latest"],
            "update_available": ha_data["data"]["update_available"]
        },
        "supervisor": {
            "version": data["data"]["version"],
            "version_latest": data["data"]["version_latest"],
            "update_available": data["data"]["update_available"]
        },
        "os": {
            "version": os_data["data"]["version"],
            "version_latest": os_data["data"]["version_latest"],
            "update_available": os_data["data"]["update_available"]
        }
    }

async def reboot_host():
    await supervisor_post("/host/reboot")
    return {"sent": True}

async def restart_ha():
    await supervisor_post("/core/restart")
    return {"sent": True}

async def shutdown_host():
    await supervisor_post("/host/shutdown")
    return {"sent": True}

async def update_core():
    await supervisor_post("/core/update")
    return {"sent": True}

async def update_supervisor():
    await supervisor_post("/supervisor/update")
    return {"sent": True}

async def update_os():
    await supervisor_post("/os/update")
    return {"sent": True}

async def create_backup(name):
    result = await supervisor_post("/backups/new/full", {"name": name})
    return result.get("data", {})
```

### 4.6 Remove Path Stripping Middleware

Delete the `StripPrefixMiddleware` class and `app.add_middleware(StripPrefixMiddleware)` — it is no longer needed. The companion no longer receives incoming HTTP requests from Harbor.

### 4.7 Remove All FastAPI Route Handlers

Remove ALL `@app.get(...)` and `@app.post(...)` route handlers EXCEPT:
- Keep `GET /health` — for local health checks

Everything else (updates, addons, backups, etc.) is now handled through the poll loop, not HTTP routes.

### 4.8 Fix: Use `lifespan` Instead of `on_event`

Replace the deprecated `@app.on_event("startup")` with the modern `lifespan` context manager:

```python
from contextlib import asynccontextmanager

@asynccontextmanager
async def lifespan(app: FastAPI):
    # Startup
    await startup()
    asyncio.create_task(poll_loop())
    yield
    # Shutdown (nothing needed)

app = FastAPI(title="Harbor Companion", version=VERSION, lifespan=lifespan)
```

## Part 6: HA Add-on Update Mechanism Fix

The issue: HA doesn't reliably detect add-on version updates from GitHub repos.

### Fix 6.1 — Companion reports version on every poll

The companion should include its version in every poll request as a header:
```
X-Companion-Version: 1.2.0
```

Harbor's poll endpoint reads this header and updates `companion_version` in the DB.

### Fix 6.2 — Harbor notifies about companion update availability

In the instance Settings tab companion section, Harbor should:
1. Show the current companion version (from DB)
2. Call `GET https://api.github.com/repos/avibarilan/Harbor/releases/latest` to get the latest release tag
3. If the latest release version > companion version → show "Update available: vX.X.X — update in Home Assistant"

### Fix 6.3 — Companion checks for its own updates

In the companion poll loop, occasionally (every 6 hours) check:
```
GET https://api.github.com/repos/avibarilan/Harbor/releases/latest
```
If a newer version exists, log: `INFO: Harbor Companion update available: v{latest}. Update via Home Assistant add-on store.`

## Part 8: Cleanup & Migration

### 8.1 Clean up stale commands
Add a background job (runs every 5 minutes) that deletes `companion_commands` rows older than 1 hour with status `done` or `error`. This prevents the table from growing forever.

Also delete commands with status `processing` older than 2 minutes (these timed out — companion died mid-execution) and set them back to `pending` so they retry.

### 8.2 Migration note for existing users
Since the companion config changes from 4 fields to 1 field, existing users must:
1. Update the companion add-on
2. Generate a new setup token in Harbor → instance Settings tab
3. Paste the token into the companion config
4. Restart the companion

Add a banner in the Harbor UI on the instance Settings tab if `companion_enabled = 1` but `companion_last_seen` is null or very old: "Companion needs to be reconfigured — generate a new setup token below."
